10 P2PSP’s Sets of Rules

10.1 IP Mulicast Set (IMS) of rules

  • The splitter sends chunks to peers using an IP multicast channel (notice that peers do not need to communicate each other).
  • Obviously, IP multicast must be avaiable!
  • The splitter runs using the --mcast ﬂag.
  • This Set is incompatible (does not make sense to use) with DBS, FNS, ACS, LRS, EMS, NTS and CIS.

10.2 Data Broadcasting Set (DBS) of rules

  • The splitter uses a Round-Robing broadcasting strategy and Unicast.
  • A peer sends the previusly received splitter’s chunk to the next peer of their list of peers only when it receives a chunk from another peer.
  • All peers contribute equally (send the same amount of data).
  • Monitor(s) peer(s) report(s) lost chunks to the splitter.
  • The peers that are already in the team insert in their list of peers to those incomming peers that send a [Hello] or a chunk of data.
  • Peers maintain a unsupportivity counter for each other peer in the team that counts the diﬀerence between the number of chunks send to and the number of chunks received from that peer. These counters are divides by two periodically.
  • DBS is the default mode and incompatible with IMS.
  • The buﬀer size must be at last as large as the number of peers of the team.

10.3 Full-cone Nat Set (FNS) of rules

  • Extends DBS.
  • Enables inbound UDP traﬃc between the splitter and the peer.
  • To achieve this, FNS-peers send a [Hello] (UDP) message to the splitter (using only DBS, peers send [Hello] messages to the rest of peers of the team when they arrive, but not to the splitter).

10.4 Adaptive Chunk-rate Set (ACS)

  • Extends DBS.
  • Peers can send a diﬀerent amount of data, depending on their estimated capacity.
  • For every peer, the splitter gathers statistics from the monitor(s) peer(s) about the lost chunk ratio and estimates a capacity.
  • The unsupportivity counters threshold must be tuned carefully (they control when a peer decides to remove from his list of peer to other peer – P1 could remove P3 –).

10.5 Lost chunks Recovery Set (LRS)

  • Extends DBS.
  • Retransmit massively lost chunks (those that have not arrived to any monitor peer).
  • Lost chunks are transmitted by at leas one monitor peer.

10.6 End-point Masquerading Set (EMS)

  • Extends FNS.
  • Allows that two or more peers can run behind the same full-cone NAT (in other words, that are in the same private network).
  • In progress ... (still not implemented :-).

10.7 NAT Traversal Set (NTS)

  • Extends FNS (which inherits from DBS).
  • FNS can connect an new peer Pn+1 but only in the case the rest of peers of the team are not behind a restricted-cone/symmetric NAT.1
  • If a peer Pn of the team is behind a restricted-cone NAT2 :
    1. When Pn+1 is joining the team, the splitter sends to Pn a [say hello to Pn+1] message.
    2. Pn says [hello] to Pn+1. This “opens” a port in the restricted-cone NAT of Pn for the incomming traﬃc from Pn+1.
  • If a peer Pn of the team is behind a symmetric NAT3 , and supposing the NAT of Pn uses the ports X,X + 1,,X + |T| where, X is the port assigned for the NAT to talk with the splitter and |T| is the current number of peers in the team T):
    1. When Pn+1 is joining the team, the splitter sends to Pn a [say hello to X + #Pn+1] message, where #Pn+1 is the position of Pn+1 in the list of peers of the splitter (all peers of the team must use the same list!!!).
    2. Pn says [hello] to Pn+1.
    3. When S sends his list of peers to Pn+1, he indicates that Pn is listening in X + #Pn+1.
  • In progress ...
  • If this tecnhique does not work, we can create a local team and connect it to an external peer or source.

10.9 Content Integrity Set (CIS)

  • Extends DBS (it does not make sense in a IP multicast scenario).
  • Rules against the poisoning of the stream, DoS (Denial of Service) attacks, etc.
  • Basically, we are planning to use a set of trusted peers that when detect an attack, they report the attacker to the splitter that will remove it from the team (trusted peers are unknown for the rest of peers of the team).
  • In progress ...

10.10 Data Privacy Set (DPS)

  • To implement, for example, pay-per-view services.
  • Needs DBS or IMS (although this last alternative would waste a lot of bandwidth because the encrypted media would reach all peers, even those that are not allowed to play the media).
  • It should rely on media encription techniques.
  • In progress ...
